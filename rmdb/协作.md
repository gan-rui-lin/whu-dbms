# RMDB 团队协作规范

本文规定团队完成各个 Stage 时必须遵循的流程。目标是让每个阶段的实现、验证、知识沉淀和 Git 历史都可以由其他成员理解并复现。

标准流程如下：

```text
阅读 stagex.md
→ 拆解任务并实现
→ 本地构建与测试
→ 验证可复现测试产物
→ 线上评测
→ 编写 stagex_guide.md
→ 创建阶段 commit
```

其中 `x` 代表阶段编号，例如 Stage 2 对应 `stage2.md` 和 `stage2_guide.md`。

## 1. 阶段准入

### 1.1 先阅读任务说明

开始 Stage X 前，负责人必须完整阅读 `stagex.md`，并确认：

- 需要实现的接口及其返回语义。
- 禁止修改或删除的公开接口、数据结构和字段。
- 涉及的模块及上下游依赖。
- 本阶段测试覆盖的功能、边界和并发要求。
- 本阶段明确不处理的内容。

`stagex.md` 是任务约束，不应在实现过程中被随意改写。调查过程中发现的理解、尝试和失败可以先记录在任务文档末尾或个人笔记中，线上通过后再整理进 `stagex_guide.md`。

### 1.2 明确负责人和修改范围

每个阶段或子模块必须有明确负责人。多人并行时，应提前划分文件和接口边界，避免同时修改同一组核心文件。

负责人开始工作前应执行：

```bash
git branch --show-current
git status --short
git pull --ff-only
```

如果工作树已有其他成员未提交的修改，不得覆盖、回退或顺手格式化这些内容。无法安全避开时，应先与修改者协调。

### 1.3 控制变更范围

- 不修改任务要求保持不变的公开接口。
- 不把其他 Stage 的实现混入当前阶段。
- 不进行与任务无关的大范围格式化、重命名或目录调整。
- 必要的公共构建修复可以纳入提交，但必须在 guide 和 commit 中说明原因。
- 每个新依赖都应说明用途；能使用项目现有依赖时，不额外引入库。

## 2. 本地实现与验证

### 2.1 构建

工具链安装和完整构建方法见 `stage0.md`。从 `rmdb/` 目录配置项目：

```bash
cmake -S . -B build
```

构建当前阶段相关目标。例如 Stage 1：

```bash
cmake --build build --target unit_test -j"$(nproc)"
```

阶段测试通过后，还必须执行一次全量构建，防止局部修改破坏其他模块：

```bash
cmake --build build -j"$(nproc)"
```

### 2.2 运行已有测试

应先运行当前阶段的定向测试，再运行完整回归测试：

```bash
# 示例：Stage 1 定向测试
./build/bin/unit_test --gtest_filter='BufferPoolManagerTest.*'

# 完整 Stage 1 回归
./build/bin/unit_test
```

并发测试至少需要重复运行，不能只依赖单次通过：

```bash
./build/bin/unit_test --gtest_filter='*ConcurrencyTest*' --gtest_repeat=100 --gtest_break_on_failure
```

当前 `unit_test` 没有注册到 CTest，因此只执行 `ctest` 不能代表 Stage 1 测试通过。具体说明见 `stage0.md`。

### 2.3 补充自己的测试

框架测试未覆盖关键边界时，负责人必须补充最小测试。测试应优先通过公开接口验证行为，并覆盖：

- 正常路径和失败路径。
- 边界值及空状态。
- 状态转换前后的不变量。
- 需要并发安全的共享状态。
- 线上失败所暴露的最小复现。

线上评测曾经失败的行为必须转化为本地回归测试，防止同一问题再次出现。

## 3. 测试产物

### 3.1 必须真实生成

本地测试应实际读写数据库文件，并产生类似以下目录的运行产物：

```text
BufferPoolManagerTest_db/
├── basic
└── concurrency
```

这些文件可以证明测试走过了磁盘、缓冲池和记录层的真实数据路径，而不只是验证内存中的返回值。

### 3.2 必须可以复现

测试产物本身默认不提交，但必须能通过 guide 中记录的命令重新生成。至少记录：

- 构建命令。
- 测试命令及必要的 filter。
- 运行测试时的工作目录。
- 预期生成的文件或目录。
- 测试通过时的关键输出。

例如：

```bash
cd build
./bin/unit_test --gtest_filter='BufferPoolManagerTest.*'
find . -maxdepth 2 -type d -name '*Test_db' -print
```

### 3.3 默认不提交

以下内容默认不得加入新的阶段提交：

- `build/` 和其他编译产物。
- `BufferPoolManagerTest_db/` 等测试数据库。
- `db.log`、调试日志、core dump。
- 临时脚本、IDE 缓存和本机配置。
- 密码、token、私钥或其他敏感信息。

提交前执行：

```bash
git status --short
git diff --check
git diff --cached
```

现有版本库中已经跟踪的历史构建产物属于遗留问题，不应作为后续继续提交同类文件的理由。

## 4. 线上评测

### 4.1 提交前

只有满足以下条件才进行线上评测：

- 当前阶段目标构建成功。
- 当前阶段相关测试全部通过。
- 完整本地回归通过。
- 并发测试已经重复运行。
- 工作树中没有意外文件或无关修改。

### 4.2 线上失败

线上测试失败时，不得将阶段标记为完成，也不得创建“Stage 完成”提交。

必须记录：

- 评测时间、得分、通过和失败用例数。
- 失败用例名称及完整断言。
- 当前实现所依赖的假设。
- 最小复现步骤。
- 最终根因和修复方式。

修复后必须先增加或更新本地回归测试，再重新执行本地构建、定向测试、完整回归和线上评测。

不要因为失败发生在并发测试中就直接假定是锁问题；应先根据断言还原它验证的数据和状态转换。

### 4.3 线上通过

线上通过后，应记录：

```text
Stage：
评测时间：
得分：
通过用例数：
失败用例数：0
对应代码状态或提交前 SHA：
```

记录完成后，进入 guide 编写和最终提交阶段。

## 5. 编写 stagex_guide.md

`stagex_guide.md` 只在线上测试通过后撰写。这样 guide 记录的是经过验证的最终理解，而不是尚未证实的猜测。

Guide 至少应包含：

- 本阶段主要类、数据结构及职责边界。
- 关键数据流和模块调用关系。
- 必须保持的核心不变量。
- 重要接口和底层能力。
- 本地构建、测试及产物复现方法。
- 实现过程中出现的失败假设、线上问题与修复。
- 对下一阶段有影响的约束或遗留问题。

Guide 不应逐段复制 `stagex.md`，也不应写成代码逐行翻译。它应帮助其他成员快速建立该阶段的整体心智模型。

## 6. Commit 规则

每个线上测试通过的 Stage 至少创建一个明确的完成提交。创建顺序为：

```text
线上测试通过
→ 记录评测结果
→ 完成 stagex_guide.md
→ 检查变更范围
→ 创建 commit
```

建议提交标题：

```text
stageX: complete <scope>
```

例如：

```text
stage1: complete storage and record managers
```

提交内容应包括：

- 本阶段实现。
- 新增或修正的测试源码。
- `stagex_guide.md`。
- 必要且有说明的构建修复。

提交前逐项检查：

```bash
git status --short
git diff --check
git diff
git diff --cached
```

提交后记录 SHA，并确认提交内容：

```bash
git log -1 --oneline
git show --stat --oneline HEAD
```

阶段提交必须能够在干净环境中重新构建和运行测试。禁止使用 `--no-verify` 绕过已有检查，禁止在未沟通的情况下重写其他成员已经使用的提交历史。

## 7. 团队沟通要求

- 开始阶段或子模块前，说明负责人、修改范围和预计依赖。
- 发现公开接口需要调整时，先与所有受影响成员确认。
- 遇到阻塞时提供命令、错误输出和已经尝试过的方法，不只描述“无法运行”。
- 线上失败后及时共享完整报告，避免其他成员重复踩坑。
- 交接时说明当前分支、最后一个有效 commit、已通过测试、未解决问题和复现命令。
- Code Review 重点检查接口语义、状态不变量、资源释放、并发边界和测试覆盖，而不只检查代码风格。

## 8. Definition of Done

只有下面所有项目均完成，Stage X 才算完成：

- [ ] 已完整阅读并完成 `stagex.md` 的要求。
- [ ] 负责人和修改范围清晰，没有覆盖其他成员的工作。
- [ ] 当前阶段目标构建成功。
- [ ] 阶段相关本地测试全部通过。
- [ ] 完整回归测试通过。
- [ ] 并发测试已重复运行并通过。
- [ ] 已生成真实测试数据产物。
- [ ] Guide 中的命令可以重新生成测试产物。
- [ ] 测试数据库、日志和 build 产物没有进入新提交。
- [ ] 线上评测通过，得分、时间和用例结果已经记录。
- [ ] 线上失败曾暴露的问题已有本地回归测试。
- [ ] `stagex_guide.md` 已完成并反映最终实现。
- [ ] `git diff --check` 无新增格式问题。
- [ ] 工作树中没有敏感信息、临时文件或无关修改。
- [ ] 最终 Stage commit 已创建并记录 SHA。
- [ ] 能从干净环境重新构建并通过测试。
