# RMDB Stage X：事务、并发控制与故障恢复

`stagex_guide.md` 对应题目九到题目十一的连续实现链路：事务控制语句、基于 no-wait 的两阶段封锁协议，以及基于 WAL 的崩溃恢复。

如果把这三题放在一条执行链上，它们其实是在解决同一个问题：

```text
SQL 执行
    ↓
TransactionManager 维护事务生命周期
    ↓
LockManager 保证并发执行的可串行性
    ↓
LogManager 记录物理日志并满足 WAL
    ↓
RecoveryManager 在系统重启后恢复一致性
```

## 1. 事务模型：显式事务和隐式事务

`Transaction` 里已经把这几个关键状态拆出来了：

- `txn_mode_`：区分显式事务和单条 SQL 的隐式事务。
- `state_`：事务当前处于 `GROWING`、`SHRINKING`、`COMMITTED`、`ABORTED` 等状态。
- `write_set_`：事务期间产生的写操作，用于回滚。
- `lock_set_`：事务持有的锁，用于提交或回滚时释放。
- `prev_lsn_`：事务最后一条日志的 LSN，用于日志链回溯。

这意味着事务管理器不是只负责“提交/回滚”两个动作，而是要贯穿整个生命周期：

```text
begin
    ↓
记录 start_ts / txn_id / 初始状态
    ↓
执行 SQL，持续维护 write_set / lock_set / prev_lsn
    ↓
commit 或 abort
```

显式事务里，`begin; ... commit;` 之间的语句共用同一个 `Transaction`。
隐式事务则通常是单条语句一个事务，执行完就提交或回滚。

## 2. 题目九：begin / commit / abort

题目九的重点不是并发，而是把事务语义真正接进执行链。

### 2.1 begin

`TransactionManager::begin()` 做三件事：

1. 创建事务或复用已有事务。
2. 赋予 `txn_id` 和开始时间戳。
3. 写入 `txn_map`，便于后续统一管理。

如果启用了日志管理器，`begin` 还要写一条 `BeginLogRecord`，并把返回的 LSN 记到 `prev_lsn_`。

### 2.2 commit

`commit` 的核心顺序是：

1. 写 `CommitLogRecord`。
2. 刷新事务写入链路。
3. 释放事务持有的所有锁。
4. 把事务状态改成 `COMMITTED`。

这里最容易漏的是“先写提交日志，再释放锁，再结束事务”。
如果先释放锁再补日志，崩溃恢复时就会失去提交边界。

### 2.3 abort

`abort` 不是简单清空状态，而是要沿着 `write_set_` 做逆向撤销：

- `INSERT_TUPLE`：删除刚插入的记录，并同步删掉对应索引项。
- `DELETE_TUPLE`：恢复被删掉的旧记录，并同步重建索引项。
- `UPDATE_TUPLE`：把记录改回旧值，并同步修正索引项。

做完回滚之后，还要释放锁；如果有日志管理器，再写 `AbortLogRecord`。

这部分最重要的原则是：**写集合记录的是可撤销信息，而不是“已经提交的事实”**。

## 3. 题目十：两阶段封锁与 no-wait

题目十把系统推进到并发执行阶段。这里的目标是可串行化，而不是“尽量多并发”。

### 3.1 锁层级

`LockManager` 支持的主要锁模式是：

- 表级：`IS`、`IX`、`S`、`X`、`SIX`
- 记录级：共享锁和排他锁

它的设计思路是典型的多粒度封锁：

```text
表锁 约束 整张表的访问语义
    ↓
记录锁 约束 具体 tuple 的读写
```

读记录时通常先拿表级 `IS`，再拿记录级 `S`。
写记录时通常先拿表级 `IX`，再拿记录级 `X`。

### 3.2 两阶段封锁

2PL 的关键不是“加锁”，而是“加锁后不能随便再加”。

在当前实现里，`unlock()` 一旦释放锁，如果事务仍处于 `GROWING`，就会把状态切到 `SHRINKING`。
之后再请求新锁，`request_lock()` 会直接抛出异常。

所以事务的生命周期可以理解成：

```text
GROWING: 只能继续获取锁
SHRINKING: 只能释放锁，不能再申请新锁
```

### 3.3 no-wait 死锁预防

这一阶段并不等待锁，而是“冲突即回滚”。

`request_lock()` 在发现申请锁与已有授予锁不兼容时，会直接抛出 `TransactionAbortException`，原因是 `DEADLOCK_PREVENTION`。

这意味着：

- 不需要死锁检测线程。
- 不需要等待队列唤醒。
- 冲突事务会立即 abort。

这种策略简单，但代价是更激进的回滚；题目测试接受这种行为。

### 3.4 常见锁申请路径

典型路径是：

- `SELECT`：表级 `IS` + 记录级 `S`
- `INSERT` / `DELETE` / `UPDATE`：表级 `IX` + 记录级 `X`
- 扫描索引时，索引页面也要按需要维护 latch / lock 关系，避免并发修改把访问路径打断

如果某个操作没有先拿到正确层级的表锁，后面的记录锁就算拿到了，也可能违反协议。

## 4. 题目十一：WAL、redo / undo 和崩溃恢复

题目十一把前两题串起来：事务执行时要写日志，崩溃后要靠日志恢复。

### 4.1 日志类型

当前日志类型包括：

- `BEGIN`
- `COMMIT`
- `ABORT`
- `INSERT`
- `DELETE`
- `UPDATE`

其中 `INSERT / DELETE / UPDATE` 是物理日志，记录了恢复所需的旧值或新值、Rid 和表名。

这里一个很关键的点是：**索引操作要按物理层恢复，而不是只记逻辑操作**。否则重启后很容易出现基表和索引不一致。

### 4.2 WAL

`LogManager` 负责把日志写到内存缓冲区，再刷到磁盘。

恢复相关的核心约束可以概括为：

```text
先有日志，后有数据落盘
```

每条日志都会拿到一个递增的 `lsn_`，事务也会用 `prev_lsn_` 串起自己的日志链。

日志缓冲区满了会先刷盘，提交时也会把日志刷新到磁盘，确保崩溃后能找到已经发生过的修改。

### 4.3 analyze / redo / undo

`RecoveryManager` 的恢复流程很直接：

1. `analyze()`：顺序读日志，收集日志列表、已提交事务和已中止事务。
2. `redo()`：重做未落盘但应该生效的物理更新。
3. `undo()`：撤销未完成事务的操作。

这里不考虑检查点，也不考虑恢复过程中的再次崩溃，所以实现上可以从第一条日志开始顺序扫描。

日志尾部如果出现撕裂写入，应该直接忽略那条不完整记录，而不是让整个恢复过程失败。

### 4.4 恢复时的索引重建

恢复时除了修复基表，还要重建索引。

当前实现的思路是：

1. 关闭已有索引句柄。
2. 删除旧索引文件。
3. 重新创建索引。
4. 扫描基表，把每条记录重新插回索引。

这比逐条回放索引日志更直接，也更符合本题“单次崩溃恢复”的测试设定。

### 4.5 恢复结束后的收尾

undo 完成后还需要：

- 刷新页缓存。
- 重建 free page list。
- 把文件头写回磁盘。
- 清空日志文件，避免下次启动误读旧日志。

如果少做这一步，数据文件本身可能对了，但缓冲池或元数据会残留脏状态。

## 5. 输出要求

这三题的测试通常有两个敏感点：

1. `SELECT` 的表格输出格式不能变，不能改 `record_printer.h` 的打印样式。
2. 因为锁预防策略或事务回滚而需要中止时，只返回 `abort\n`，不要额外打印调试信息。

恢复测试里，单线程场景通常检查数据库目录下的 `output.txt`，多线程场景则通过客户端返回字符串判断。

## 6. 常见错误

### 6.1 只改状态，不回滚写集合

abort 之后如果没有按 `write_set_` 逆序撤销，事务表面上结束了，数据实际上还在。

### 6.2 释放锁太早

如果在写日志之前就释放锁，恢复和并发执行都会失去约束。

### 6.3 忽略 prev_lsn_

不维护 `prev_lsn_`，事务日志之间就断链了，恢复阶段很难判断事务的完整历史。

### 6.4 只恢复基表，不恢复索引

索引不一致是恢复阶段最常见的隐性错误之一。基表看起来正确，但查询走索引时结果会错。

### 6.5 把等待锁写成阻塞锁

本题要求的是 no-wait。若实现成等待队列，测试里的冲突事务就不会按预期立即返回 `abort\n`。

## 7. 推荐完成顺序

1. 先把 begin / commit / abort 的事务生命周期跑通。
2. 再接入 LockManager，确保 2PL 和 no-wait 冲突处理正确。
3. 最后补 WAL 和 RecoveryManager，把崩溃恢复、redo / undo 和索引重建接起来。

如果这三步能依次通过，stage9 到 stage11 的链路就基本稳定了。
